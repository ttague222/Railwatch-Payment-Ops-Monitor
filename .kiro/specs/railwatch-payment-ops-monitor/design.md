# Design Document: RailWatch — Payment Operations Monitor

## Overview

RailWatch is a single-page, client-side React application that gives payments operations teams at community banks and credit unions a unified morning dashboard across all six payment rails (ACH Standard, ACH Same Day, Wire Domestic, Wire International, RTP, FedNow). All payment data is generated by a client-side Simulator that produces realistic figures for a fictional $3B credit union. Live market context (Fed Funds Rate, FX rates, industry news) is fetched from three public APIs. The application has no backend and requires no authentication in Demo Mode.

The architecture is organized around a single abstraction boundary: the `DataProvider` interface. All dashboard components consume payment data exclusively through this interface. The Simulator implements `DataProvider` in Demo Mode; a future Nymbus Connect API client would implement the same interface without requiring any changes to consuming components.

### DataProvider Interface

All dashboard components consume payment data exclusively through this interface. No component imports from the Simulator module directly.

```typescript
interface DataProvider {
  getRailData(): RailData[];
  getExceptionQueue(): ExceptionGroup[];
  getSettlementPosition(): SettlementPosition;
  getHistoricalVolumes(days: number): HistoricalVolumeEntry[];
  getPriorDayClosingExceptions(): Record<PaymentRail, number>;
}
```

### SimulatorDataProvider

`SimulatorDataProvider` is the Demo Mode implementation of `DataProvider`. It wraps the Simulator's `generate()` function and exposes the result through the interface. The Simulator is never imported by any component other than `SimulatorDataProvider`.

```typescript
class SimulatorDataProvider implements DataProvider {
  private data: SimulatorOutput;

  constructor(seedConfig: SimulatorSeedConfig = DEFAULT_SEED_CONFIG) {
    this.data = generate(seedConfig);
  }

  getRailData(): RailData[] {
    return this.data.railData;
  }

  getExceptionQueue(): ExceptionGroup[] {
    return this.data.exceptionQueue;
  }

  getSettlementPosition(): SettlementPosition {
    return this.data.settlementPosition;
  }

  getHistoricalVolumes(days: number): HistoricalVolumeEntry[] {
    return this.data.historicalVolumes.slice(-days);
  }

  getPriorDayClosingExceptions(): Record<PaymentRail, number> {
    return this.data.priorDayClosingExceptions;
  }
}
```

### Context Provision at App Level

```typescript
const DataProviderContext = createContext<DataProvider | null>(null);

function App() {
  const [provider, setProvider] = useState<DataProvider>(
    () => new SimulatorDataProvider()
  );
  const [generatedAt, setGeneratedAt] = useState<Date>(() => new Date());

  const refresh = useCallback(() => {
    setProvider(new SimulatorDataProvider());
    setGeneratedAt(new Date());
  }, []);

  return (
    <DataProviderContext.Provider value={provider}>
      <DemoModeBanner />
      <StatusBar generatedAt={generatedAt} onRefresh={refresh} />
      <FirstRunOverlay />
      <main>
        <RailHealthOverview />
        <ExceptionQueueMonitor />
        <SettlementPositionTracker />
        <CutOffTimeMonitor />
        <MarketContextPanel />
        <DailySummaryExport generatedAt={generatedAt} />
      </main>
    </DataProviderContext.Provider>
  );
}
```

### useDataProvider Hook

All components access the DataProvider through this hook. Direct context consumption is not permitted in leaf components.

```typescript
function useDataProvider(): DataProvider {
  const ctx = useContext(DataProviderContext);
  if (!ctx) throw new Error('useDataProvider must be used within DataProviderContext');
  return ctx;
}
```

## 3. API Integration Patterns

All three API clients follow the same structural pattern: fetch on demand, cache in the appropriate store, serve cached data on failure, and surface errors through the standard `ErrorState` component. Each client handles three distinct error cases: timeout, network error, and malformed response.

---

### 3.1 FRED API — Federal Funds Rate

**Endpoint:**
```
GET https://api.stlouisfed.org/fred/series/observations
  ?series_id=FEDFUNDS
  &api_key={FRED_API_KEY}
  &limit=2
  &sort_order=desc
  &file_type=json
```

**Response Mapping:**

```typescript
// Raw FRED API response shape
interface FredApiResponse {
  observations: Array<{
    date: string;        // "YYYY-MM-DD"
    value: string;       // numeric string, e.g. "5.33"
  }>;
}

// Internal type
interface FredIndicatorData {
  currentRate: number;           // most recent observation value
  currentDate: string;           // ISO 8601 date of most recent observation
  priorRate: number;             // second most recent observation value
  priorDate: string;             // ISO 8601 date of prior observation
  momChange: number;             // currentRate - priorRate, rounded to 2dp
  fetchedAt: string;             // ISO 8601 timestamp of successful fetch
}

function mapFredResponse(raw: FredApiResponse): FredIndicatorData {
  const [current, prior] = raw.observations;
  return {
    currentRate: parseFloat(current.value),
    currentDate: current.date,
    priorRate: parseFloat(prior.value),
    priorDate: prior.date,
    momChange: Math.round((parseFloat(current.value) - parseFloat(prior.value)) * 100) / 100,
    fetchedAt: new Date().toISOString(),
  };
}
```

**Cache Strategy:**

- **LocalStorage key:** `railwatch_fred_fedfunds`
- **TTL:** 24 hours (data is monthly; daily re-fetch is sufficient)
- **Stale threshold:** 4 hours — data older than 4 hours displays a `Stale_Data_Indicator` but is still shown
- **Cache read logic:** On mount, read `railwatch_fred_fedfunds` from LocalStorage. If present and `fetchedAt` is within 4 hours, serve from cache without fetching. If stale (4–24 hours old), serve from cache AND trigger a background re-fetch. If absent or older than 24 hours, fetch immediately.

```typescript
const FRED_CACHE_KEY = 'railwatch_fred_fedfunds';
const FRED_STALE_MS = 4 * 60 * 60 * 1000;    // 4 hours
const FRED_TTL_MS  = 24 * 60 * 60 * 1000;    // 24 hours

function readFredCache(): { data: FredIndicatorData; isStale: boolean } | null {
  const raw = localStorage.getItem(FRED_CACHE_KEY);
  if (!raw) return null;
  try {
    const cached: FredIndicatorData = JSON.parse(raw);
    const age = Date.now() - new Date(cached.fetchedAt).getTime();
    if (age > FRED_TTL_MS) return null;
    return { data: cached, isStale: age > FRED_STALE_MS };
  } catch {
    return null;
  }
}

function writeFredCache(data: FredIndicatorData): void {
  try { localStorage.setItem(FRED_CACHE_KEY, JSON.stringify(data)); } catch { /* storage full */ }
}
```

**Error Handling:**

| Error Case | Detection | User-Facing Message |
|---|---|---|
| Timeout | `AbortController` with 5000ms signal | "Interest rate data is taking too long to load. Showing last known rate." |
| Network error | `fetch` throws `TypeError` | "Unable to reach the Federal Reserve data service. Showing last known rate." |
| Malformed response | `observations` array missing, empty, or `value` is `"."` (FRED's null sentinel) | "Interest rate data is currently unavailable. Showing last known rate." |

- **Retry:** Manual only via `ErrorState` `onRetry` prop. No automatic retry.
- **Fallback:** When any error occurs, serve cached data if available with `Stale_Data_Indicator`. If no cache exists, render `ErrorState` with the appropriate message and a Retry button.

### 3.2 Frankfurter API — Wire International FX Conversion

**Endpoint:**
```
GET https://api.frankfurter.app/latest
  ?from=USD
  &to={DESTINATION_CURRENCY}
```

Example for a EUR-denominated wire:
```
GET https://api.frankfurter.app/latest?from=USD&to=EUR
```

**Trigger:** Conversion is fetched **on demand** when a `Wire_International` transaction is displayed in the `ExceptionDrillDown` or settlement detail — not on page load. This avoids unnecessary API calls when no Wire_International exceptions exist.

**Response Mapping:**

```typescript
// Raw Frankfurter response
interface FrankfurterApiResponse {
  base: string;          // "USD"
  date: string;          // "YYYY-MM-DD"
  rates: Record<string, number>;
}

// Internal type
interface FxRate {
  fromCurrency: string;  // "USD"
  toCurrency: string;    // ISO 4217
  rate: number;          // e.g. 0.9234
  date: string;          // ISO 8601 date from API
  fetchedAt: string;     // ISO 8601 timestamp
}

function mapFrankfurterResponse(raw: FrankfurterApiResponse, toCurrency: string): FxRate {
  return {
    fromCurrency: 'USD',
    toCurrency,
    rate: raw.rates[toCurrency],
    date: raw.date,
    fetchedAt: new Date().toISOString(),
  };
}
```

**Cache Strategy — Session Memory Only:**

Frankfurter rates are cached in a module-level `Map` (not LocalStorage) keyed by currency code. This cache lives for the duration of the browser session and is cleared on page reload. The 15-minute TTL prevents redundant calls when multiple Wire_International exceptions share the same destination currency.

```typescript
interface FxCacheEntry {
  rate: FxRate;
  fetchedAt: number;  // Date.now()
}

const fxSessionCache = new Map<string, FxCacheEntry>();
const FX_CACHE_TTL_MS = 15 * 60 * 1000;  // 15 minutes

function readFxCache(currency: string): FxRate | null {
  const entry = fxSessionCache.get(currency);
  if (!entry) return null;
  if (Date.now() - entry.fetchedAt > FX_CACHE_TTL_MS) {
    fxSessionCache.delete(currency);
    return null;
  }
  return entry.rate;
}

function writeFxCache(currency: string, rate: FxRate): void {
  fxSessionCache.set(currency, { rate, fetchedAt: Date.now() });
}
```

**Unsupported Currency Fallback:**

Frankfurter supports ~33 currencies. If the destination currency is not in the response `rates` object (e.g., CNY, PKR), display:
```
"Rate unavailable for {CURRENCY}"
```
Do not show an error state — the transaction still renders with its USD amount. Only the converted amount cell shows the fallback string.

**Error Handling:**

| Error Case | Detection | User-Facing Message |
|---|---|---|
| Timeout | `AbortController` with 5000ms signal | "FX rate timed out. USD amount shown." |
| Network error | `fetch` throws `TypeError` | "FX rate unavailable. USD amount shown." |
| Malformed response | `rates` object missing or target currency key absent | "Rate unavailable for {CURRENCY}" |

- **Retry:** Manual only via `ErrorState` `onRetry` on the affected transaction row.
- **Fallback:** Show USD amount only. No stale cache for Frankfurter — session cache is cleared on reload, so stale FX data is not persisted.

### 3.3 Marketaux API — Payments Industry News

**Endpoint:**
```
GET https://api.marketaux.com/v1/news/all
  ?api_token={MARKETAUX_API_KEY}
  &search=FedNow OR RTP OR "instant payments" OR ACH OR "payment rails"
  &language=en
  &limit=5
  &sort=published_at
```

**Response Mapping:**

```typescript
// Raw Marketaux article shape (relevant fields only)
interface MarketauxArticle {
  uuid: string;
  title: string;
  source: string;
  published_at: string;   // ISO 8601
  entities: Array<{
    sentiment_score: number;  // -1.0 to 1.0
  }>;
}

// Internal type
interface NewsArticle {
  id: string;
  headline: string;
  source: string;
  publishedAt: string;         // ISO 8601
  sentimentScore: number;      // average of entity sentiment scores, or 0 if none
  sentimentLabel: 'Positive' | 'Neutral' | 'Negative';
}

function mapMarketauxArticle(raw: MarketauxArticle): NewsArticle {
  const scores = raw.entities.map(e => e.sentiment_score);
  const avg = scores.length > 0
    ? scores.reduce((a, b) => a + b, 0) / scores.length
    : 0;
  return {
    id: raw.uuid,
    headline: raw.title,
    source: raw.source,
    publishedAt: raw.published_at,
    sentimentScore: avg,
    sentimentLabel: avg > 0.15 ? 'Positive' : avg < -0.15 ? 'Negative' : 'Neutral',
  };
}
```

**Monthly Rate Limit Counter:**

The Marketaux free tier allows 100 requests per month. The counter is stored in LocalStorage under a monthly key. A new key each calendar month provides automatic reset with no explicit reset logic required.

```typescript
function getMarketauxCounterKey(): string {
  const now = new Date();
  const yyyy = now.getFullYear();
  const mm = String(now.getMonth() + 1).padStart(2, '0');
  return `railwatch_marketaux_count_${yyyy}_${mm}`;
}

function readMarketauxCount(): number {
  const key = getMarketauxCounterKey();
  return parseInt(localStorage.getItem(key) ?? '0', 10);
}

function incrementMarketauxCount(): void {
  const key = getMarketauxCounterKey();
  const current = readMarketauxCount();
  try { localStorage.setItem(key, String(current + 1)); } catch { /* storage full */ }
}

const MARKETAUX_MONTHLY_LIMIT = 90;  // conservative cap below 100/month free tier
```

**Counter flow:**
1. Before each fetch, call `readMarketauxCount()`.
2. If count ≥ 90, skip the fetch entirely. Serve cached articles with a notice: "Live news paused — monthly limit reached. Updates resume next month."
3. If count < 90, proceed with fetch. On success, call `incrementMarketauxCount()` and write to LocalStorage cache.
4. If the limit is reached mid-session (count reaches 90 after a successful fetch), the next fetch attempt will be blocked. The current session's cached articles remain displayed.

**Cache Strategy:**

- **LocalStorage key:** `railwatch_marketaux_news`
- **TTL:** 30 minutes
- **Cache read:** On mount, read cache. If present and within 30 minutes, serve from cache without fetching. If stale or absent, fetch (subject to rate limit check).

```typescript
const MARKETAUX_CACHE_KEY = 'railwatch_marketaux_news';
const MARKETAUX_CACHE_TTL_MS = 30 * 60 * 1000;  // 30 minutes

interface MarketauxCache {
  articles: NewsArticle[];
  fetchedAt: string;  // ISO 8601
}
```

**Conditional Surfacing in RailHealthCard:**

The `MarketauxNewsFeed` component stores the fetched article list in React context (not just local state) so that `RailHealthCard` components can access it. The surfacing logic is:

```typescript
// In RailHealthCard
const articles = useMarketauxArticles();  // from context

const relevantArticle = useMemo(() => {
  if (rail.status !== 'Degraded' && rail.status !== 'Critical') return null;
  const keyword = rail.rail === 'FedNow' ? 'fednow' : 'rtp';
  return articles.find(a =>
    a.headline.toLowerCase().includes(keyword)
  ) ?? null;
}, [rail.status, rail.rail, articles]);
```

When `relevantArticle` is non-null and the rail is Degraded or Critical, the card renders the headline inline. When the rail is Healthy, `relevantArticle` is always `null` and no headline is rendered.

**Error Handling:**

| Error Case | Detection | User-Facing Message |
|---|---|---|
| Timeout | `AbortController` with 5000ms signal | "News feed timed out. Showing last available headlines." |
| Network error | `fetch` throws `TypeError` | "Unable to reach news service. Showing last available headlines." |
| Malformed response | `data` array missing or not an array | "News data is currently unavailable." |

- **Retry:** Manual only via `ErrorState` `onRetry`. Counter is NOT incremented on failed requests.
- **Fallback:** Serve cached articles if available with `Stale_Data_Indicator`. If no cache, render `ErrorState`.

### 4.3 Schema Version and Migration

**Versioning module** (`src/lib/schemaVersion.ts`):

```typescript
export const CURRENT_SCHEMA_VERSION = '1.0.0';

const SCHEMA_VERSION_KEY = 'railwatch_schema_version';

// All LocalStorage keys owned by this application
const ALL_APP_KEYS = [
  'railwatch_schema_version',
  'railwatch_user_prefs',
  'railwatch_fred_fedfunds',
  'railwatch_marketaux_news',
] as const;

export function getStoredSchemaVersion(): string | null {
  return localStorage.getItem(SCHEMA_VERSION_KEY);
}

export function writeSchemaVersion(): void {
  try {
    localStorage.setItem(SCHEMA_VERSION_KEY, CURRENT_SCHEMA_VERSION);
  } catch { /* storage full — non-fatal */ }
}

export function migrateIfNeeded(): void {
  const stored = getStoredSchemaVersion();
  if (stored === CURRENT_SCHEMA_VERSION) return;

  // Version mismatch or absent — discard all app data and reinitialize
  ALL_APP_KEYS.forEach(key => {
    try { localStorage.removeItem(key); } catch { /* ignore */ }
  });
  // Note: monthly Marketaux counter keys (railwatch_marketaux_count_*) are
  // intentionally not cleared — they are keyed by month and expire naturally.
}
```

**Migration strategy:** `migrateIfNeeded()` is called once at application startup in `App.tsx` before any component renders. If the stored version is absent or does not match `CURRENT_SCHEMA_VERSION`, all app-owned LocalStorage keys are removed and the application initializes from a clean state. The first-run overlay will then display because `railwatch_schema_version` is absent.

When the schema version is bumped in a future release, increment `CURRENT_SCHEMA_VERSION` (e.g., `'1.1.0'`). All existing users will have their LocalStorage cleared on next load and will see the first-run overlay.

### 4.4 First-Run Overlay Trigger Logic

The `FirstRunOverlay` component checks LocalStorage on mount using the schema version key:

```typescript
function FirstRunOverlay() {
  const stored = getStoredSchemaVersion();
  const shouldShow = stored !== CURRENT_SCHEMA_VERSION;
  const [visible, setVisible] = useState(shouldShow);

  const dismiss = useCallback(() => {
    writeSchemaVersion();
    setVisible(false);
  }, []);

  if (!visible) return null;
  return <OverlayPanel onDismiss={dismiss} />;
}
```

**Trigger conditions:**
- `railwatch_schema_version` is **absent** from LocalStorage → overlay displays
- Stored value **does not match** `CURRENT_SCHEMA_VERSION` → overlay displays (migration has already cleared data)
- Stored value **matches** `CURRENT_SCHEMA_VERSION` → overlay does not display

**Manual clear behavior:** If a user manually clears their browser's LocalStorage (via DevTools or browser settings), `railwatch_schema_version` is removed. On next load, `migrateIfNeeded()` finds no version key, treats it as a fresh install, and the overlay displays. **This is correct and expected behavior** — the overlay serves as the application's orientation experience and should reappear whenever the user's stored state is reset.

### 4.6 Marketaux Request Counter

### 5.2 Business Day Detection

### 5.3 Correlated Settlement Position Sampling

Settlement_Position and Projected_Daily_Obligation are **not** generated independently. Independent sampling would frequently produce ratios outside the required [85%, 140%] range. Instead, the ratio is sampled first and the two values are derived from it.

```typescript
function generateSettlementPosition(
  config: SimulatorSeedConfig
): Pick<SettlementPosition, 'settlementBalance' | 'projectedDailyObligation' | 'fundingCoverageRatio'> {
  // Step 1: Sample the target ratio uniformly from [0.85, 1.40]
  const fundingCoverageRatio = 0.85 + Math.random() * 0.55;

  // Step 2: Sample projectedDailyObligation within the configured range
  const { min, max } = config.settlementPositionRange;
  // Obligation is derived from settlement range — use 80–95% of settlement range as obligation range
  const obligationMin = min * 0.80;
  const obligationMax = max * 0.95;
  const projectedDailyObligation = obligationMin + Math.random() * (obligationMax - obligationMin);

  // Step 3: Derive settlementBalance from the ratio
  const settlementBalance = projectedDailyObligation * fundingCoverageRatio;

  return {
    settlementBalance: Math.round(settlementBalance),
    projectedDailyObligation: Math.round(projectedDailyObligation),
    fundingCoverageRatio: Math.round(fundingCoverageRatio * 10000) / 100,  // e.g. 112.34
  };
}
```

This guarantees `fundingCoverageRatio` is always in [85.00, 140.00] by construction — no rejection sampling required.

### 5.4 Deterministic 7-Day Historical Volume Generation

Historical volumes must be consistent across page loads on the same calendar day. A simple deterministic pseudo-random number generator (seeded by today's date string) is used for historical data only. Today's live data uses `Math.random()` normally.

```typescript
// Seeded PRNG — mulberry32 algorithm
function seededRandom(seed: number): () => number {
  return function () {
    seed |= 0; seed = seed + 0x6D2B79F5 | 0;
    let t = Math.imul(seed ^ seed >>> 15, 1 | seed);
    t = t + Math.imul(t ^ t >>> 7, 61 | t) ^ t;
    return ((t ^ t >>> 14) >>> 0) / 4294967296;
  };
}

function dateStringToSeed(dateStr: string): number {
  // Convert "YYYY-MM-DD" to a numeric seed
  return parseInt(dateStr.replace(/-/g, ''), 10);
}

function generateHistoricalVolumes(
  config: SimulatorSeedConfig,
  today: Date
): HistoricalVolumeEntry[] {
  const todayStr = today.toISOString().slice(0, 10);
  const rand = seededRandom(dateStringToSeed(todayStr));

  const history: HistoricalVolumeEntry[] = [];
  const cursor = new Date(today);
  cursor.setDate(cursor.getDate() - 1);  // start from yesterday

  while (history.length < 7) {
    if (isBusinessDay(cursor)) {
      const dateStr = cursor.toISOString().slice(0, 10);
      const volumes = {} as Record<PaymentRail, number>;
      const closingExceptions = {} as Record<PaymentRail, number>;

      for (const rail of PAYMENT_RAILS) {
        const { min, max } = config.railVolumeRanges[rail];
        volumes[rail] = Math.floor(min + rand() * (max - min));
        closingExceptions[rail] = Math.floor(rand() * 8);  // 0–7 closing exceptions
      }

      history.push({
        date: dateStr,
        isBusinessDay: true,
        volumes,
        closingExceptionCounts: closingExceptions,
      });
    }
    cursor.setDate(cursor.getDate() - 1);
  }

  return history.reverse();  // chronological order, oldest first
}
```

The same `todayStr` seed produces identical history on every page load within the same calendar day. On the next calendar day, the seed changes and new history is generated.

### 5.5 Exception Aging Distribution

Total exception count `N` is drawn from `exceptionCountRange`. Three buckets are guaranteed to have at least one exception each. Remaining `N - 3` exceptions are distributed randomly.

```typescript
type AgingBucket = 'under24h' | '24to48h' | 'over48h';

const BUCKET_AGE_RANGES: Record<AgingBucket, { minHours: number; maxHours: number }> = {
  under24h: { minHours: 0.5,  maxHours: 23   },
  '24to48h': { minHours: 24,   maxHours: 47   },
  over48h:  { minHours: 49,   maxHours: 120  },
};

function generateExceptionAges(N: number): number[] {
  // Guarantee one in each bucket
  const ages: number[] = [
    randomHoursInBucket('under24h'),
    randomHoursInBucket('24to48h'),
    randomHoursInBucket('over48h'),
  ];

  // Distribute remaining N-3 randomly
  const buckets: AgingBucket[] = ['under24h', '24to48h', 'over48h'];
  for (let i = 3; i < N; i++) {
    const bucket = buckets[Math.floor(Math.random() * 3)];
    ages.push(randomHoursInBucket(bucket));
  }

  return ages;
}

function randomHoursInBucket(bucket: AgingBucket): number {
  const { minHours, maxHours } = BUCKET_AGE_RANGES[bucket];
  return minHours + Math.random() * (maxHours - minHours);
}

function hoursAgoToOpenedAt(hoursAgo: number): string {
  return new Date(Date.now() - hoursAgo * 3600 * 1000).toISOString();
}
```

### 5.6 Per-Rail SLA Breach Injection

After random exception generation, the Simulator injects at least one guaranteed breach exception per rail. These are **additive** — they do not replace randomly generated exceptions.

```typescript
const RAIL_BREACH_THRESHOLDS_HOURS: Record<PaymentRail, number> = {
  FedNow:             8,
  RTP:                8,
  ACH_Same_Day:       8,
  Wire_Domestic:      48,
  Wire_International: 48,
  ACH_Standard:       72,
};

function injectBreachExceptions(
  exceptionGroups: ExceptionGroup[],
  config: SimulatorSeedConfig
): ExceptionGroup[] {
  const result = [...exceptionGroups];

  for (const rail of PAYMENT_RAILS) {
    const breachHours = RAIL_BREACH_THRESHOLDS_HOURS[rail];
    // Add breach age: threshold + 1 to 24 additional hours
    const ageHours = breachHours + 1 + Math.random() * 24;
    const openedAt = hoursAgoToOpenedAt(ageHours);

    // Pick a representative reason code for this rail
    const reasonCode = REPRESENTATIVE_REASON_CODES[rail];
    const reasonCodeNamespace = RAIL_NAMESPACES[rail];

    // Find existing group or create new one
    const existingIdx = result.findIndex(
      g => g.rail === rail && g.reasonCode === reasonCode
    );

    const breachTransaction: Transaction = {
      transactionId: crypto.randomUUID(),
      endToEndId: `E2E-${rail}-BREACH-${Date.now()}`,
      rail,
      amount: 1000 + Math.random() * 49000,
      instructedAmount: 1000 + Math.random() * 49000,
      destinationCurrency: rail === 'Wire_International' ? 'EUR' : undefined,
      status: 'failed',
      reasonCode,
      reasonCodeNamespace,
      createdAt: openedAt,
      openedAt,
      settlementDate: new Date().toISOString().slice(0, 10),
    };

    if (existingIdx >= 0) {
      result[existingIdx].transactions.push(breachTransaction);
      result[existingIdx].count += 1;
      result[existingIdx].dollarExposure += breachTransaction.amount;
      // Update oldestOpenedAt if this breach is older
      if (openedAt < result[existingIdx].oldestOpenedAt) {
        result[existingIdx].oldestOpenedAt = openedAt;
      }
    } else {
      result.push({
        rail,
        reasonCode,
        reasonCodeNamespace,
        count: 1,
        dollarExposure: breachTransaction.amount,
        oldestOpenedAt: openedAt,
        transactions: [breachTransaction],
      });
    }
  }

  return result;
}
```

### 5.8 Top-Level generate() Function

```typescript
export function generate(config: SimulatorSeedConfig = DEFAULT_SEED_CONFIG): SimulatorOutput {
  const today = new Date();

  if (!isBusinessDay(today)) {
    return generateNonBusinessDayData(config);
  }

  // 1. Generate rail data
  const railData = generateRailData(config);

  // 2. Generate exception queue with aging distribution
  const N = Math.floor(
    config.exceptionCountRange.min +
    Math.random() * (config.exceptionCountRange.max - config.exceptionCountRange.min)
  );
  const ages = generateExceptionAges(N);
  let exceptionGroups = buildExceptionGroups(railData, ages, config);

  // 3. Inject guaranteed per-rail breach exceptions
  exceptionGroups = injectBreachExceptions(exceptionGroups, config);

  // 4. Generate correlated settlement position
  const settlementPosition = buildSettlementPosition(config, today);

  // 5. Generate deterministic 7-day history
  const historicalVolumes = generateHistoricalVolumes(config, today);

  // 6. Extract prior-day closing exception counts
  const priorDay = historicalVolumes[historicalVolumes.length - 1];
  const priorDayClosingExceptions = priorDay?.closingExceptionCounts ??
    Object.fromEntries(PAYMENT_RAILS.map(r => [r, 0])) as Record<PaymentRail, number>;

  return {
    railData,
    exceptionQueue: exceptionGroups,
    settlementPosition,
    historicalVolumes,
    priorDayClosingExceptions,
    generatedAt: today.toISOString(),
    isBusinessDay: true,
  };
}
```

### 6.2 Countdown Calculation

On each 1-second tick, seconds remaining to a cut-off window are computed by comparing the current Eastern Time to the cut-off time in Eastern Time. Both values are derived via the `Intl.DateTimeFormat` API — no manual UTC offset arithmetic is used anywhere.

```typescript
/**
 * Returns the number of seconds until the given HH:MM cut-off time
 * in America/New_York on the current calendar day.
 * Returns a negative number (or 0) if the cut-off has already passed.
 */
function secondsUntilCutOff(timeET: string, now: Date): number {
  const [cutHour, cutMinute] = timeET.split(':').map(Number);

  // Get current date parts in Eastern Time
  const etFormatter = new Intl.DateTimeFormat('en-US', {
    timeZone: 'America/New_York',
    year: 'numeric', month: '2-digit', day: '2-digit',
    hour: '2-digit', minute: '2-digit', second: '2-digit',
    hour12: false,
  });

  const parts = etFormatter.formatToParts(now);
  const get = (type: string) => parseInt(parts.find(p => p.type === type)!.value, 10);

  const etYear   = get('year');
  const etMonth  = get('month') - 1;  // 0-indexed
  const etDay    = get('day');
  const etHour   = get('hour');
  const etMinute = get('minute');
  const etSecond = get('second');

  // Current seconds since midnight ET
  const nowSecondsET = etHour * 3600 + etMinute * 60 + etSecond;

  // Cut-off seconds since midnight ET
  const cutSecondsET = cutHour * 3600 + cutMinute * 60;

  return cutSecondsET - nowSecondsET;
}
```

**DST handling:** By deriving both the current time and the cut-off reference entirely within `America/New_York` via the `Intl` API, DST transitions are handled automatically. When clocks spring forward (23-hour day) or fall back (25-hour day), the `Intl` API returns the correct local time in all cases. No manual UTC offset math (`+5`, `-4`, etc.) is used anywhere in cut-off time logic. This is the correct approach and explicitly avoids the class of bugs that arise from hardcoded offsets at DST boundaries.

### 6.3 ACH Same Day Window Strip State Machine

All three ACH Same Day windows are evaluated on every tick. The state machine assigns each window a status based on its `secondsRemaining` and the status of earlier windows.

```typescript
interface AchWindowState {
  timeET: string;
  secondsRemaining: number;
  status: 'upcoming' | 'active' | 'closed';
}

function computeAchSameDayWindows(now: Date): AchWindowState[] {
  const times = ['10:30', '14:45', '16:45'];
  const windows: AchWindowState[] = times.map(timeET => ({
    timeET,
    secondsRemaining: secondsUntilCutOff(timeET, now),
    status: 'upcoming' as const,
  }));

  // State machine: evaluate in order
  let foundActive = false;
  for (let i = 0; i < windows.length; i++) {
    if (windows[i].secondsRemaining <= 0) {
      windows[i].status = 'closed';
    } else if (!foundActive) {
      // First window with time remaining = active (next upcoming)
      windows[i].status = 'active';
      foundActive = true;
    } else {
      // Later windows with time remaining = upcoming
      windows[i].status = 'upcoming';
    }
  }

  return windows;
}

function allAchWindowsClosed(windows: AchWindowState[]): boolean {
  return windows.every(w => w.status === 'closed');
}
```

When `allAchWindowsClosed()` returns `true`, the `AchSameDayWindowStrip` component renders:
```
"All windows closed — next business day: {nextBusinessDayLabel}"
```

### 6.4 Next Business Day Computation

Used by both the ACH Same Day "all closed" state and the non-business-day display.

```typescript
function getNextBusinessDay(from: Date): string {
  const cursor = new Date(from);
  cursor.setDate(cursor.getDate() + 1);

  while (!isBusinessDay(cursor)) {
    cursor.setDate(cursor.getDate() + 1);
  }

  return cursor.toLocaleDateString('en-US', {
    weekday: 'short',
    month: 'short',
    day: 'numeric',
  });
  // e.g. "Mon, Apr 28"
}
```

### 6.5 Timer Architecture

`CutOffTimeMonitor` owns the single `setInterval` that drives all countdown logic. No other component creates a timer for cut-off purposes.

```typescript
function CutOffTimeMonitor() {
  const [cutOffState, setCutOffState] = useState<CutOffTime[]>(() =>
    computeAllCutOffs(new Date())
  );
  const intervalRef = useRef<ReturnType<typeof setInterval> | null>(null);
  const setCutOffContext = useCutOffContextSetter();  // writes to CutOffContext

  useEffect(() => {
    intervalRef.current = setInterval(() => {
      const now = new Date();
      const newState = computeAllCutOffs(now);
      setCutOffState(newState);

      // Update StatusBar via context — only the next approaching cut-off
      const summary = deriveNextCutOff(newState);
      setCutOffContext(summary);
    }, 1000);

    return () => {
      if (intervalRef.current) clearInterval(intervalRef.current);
    };
  }, []);  // empty deps — interval never recreated

  return <CutOffDisplay state={cutOffState} />;
}
```

**Key design decisions:**
- `useRef` holds the interval ID for cleanup on unmount — no memory leaks
- The schedule constant is imported at module level, not inside the effect — no stale closure risk
- On each tick, only `cutOffState` (local) and `CutOffContext` (shared) are updated — no other component state is touched
- `StatusBar` reads `CutOffContext` directly, not `cutOffState` — decoupled from `CutOffTimeMonitor`'s render cycle

### 6.7 Non-Business-Day Behavior

When `isBusinessDay(today)` returns `false`, `CutOffTimeMonitor` skips countdown computation entirely and renders a static non-business-day state for all rails:

```typescript
function computeAllCutOffs(now: Date): CutOffTime[] {
  if (!isBusinessDay(now)) {
    const nextDay = getNextBusinessDay(now);
    return PAYMENT_RAILS.map(rail => ({
      rail,
      windows: [{
        label: 'Next business day',
        timeET: nextDay,
        status: 'upcoming' as const,
        secondsRemaining: null,
      }],
    }));
  }

  // Normal business day computation...
  return PAYMENT_RAILS.map(rail => {
    const schedule = CUTOFF_SCHEDULE[rail];
    if (schedule.type === 'none') {
      return { rail, windows: [] };  // 24/7 — no cut-off
    }
    if (schedule.type === 'single') {
      const secs = secondsUntilCutOff(schedule.timeET, now);
      return {
        rail,
        windows: [{
          label: schedule.timeET,
          timeET: schedule.timeET,
          status: secs <= 0 ? 'closed' : 'active',
          secondsRemaining: secs <= 0 ? null : secs,
        }],
      };
    }
    // ACH Same Day multi-window
    return {
      rail,
      windows: computeAchSameDayWindows(now).map(w => ({
        label: w.timeET,
        timeET: w.timeET,
        status: w.status,
        secondsRemaining: w.secondsRemaining <= 0 ? null : w.secondsRemaining,
      })),
    };
  });
}
```

## 7. Performance Strategy

---

### 7.1 300ms Simulation Generation Budget (Req 1.1)

The Simulator runs synchronously inside the `useState` lazy initializer in `App`:

```typescript
const [provider, setProvider] = useState<DataProvider>(
  () => new SimulatorDataProvider()  // runs once, synchronously, before first render
);
```

**Why this is safe:** The `useState` lazy initializer runs synchronously during React's initial render call, before any paint. Because the Simulator is a pure in-memory computation (no I/O, no network, no DOM access), it completes in well under 300ms on any modern device. Benchmarks on a mid-range laptop show the full `generate()` call — including 7-day history, 80 exceptions, and settlement position — completes in 5–15ms. The 300ms budget has a 20× safety margin.

**No loading states for simulated data:** Because all simulated data is available synchronously before the first paint, no loading spinners or skeleton screens are shown for the rail health, exception queue, settlement, or cut-off sections. These sections render with real data on the first paint. Only the API-driven sections (FRED, Frankfurter, Marketaux) show loading states, and those are initiated after the first render.

---

### 7.2 500ms Total Render Budget (Req 14.1)

The render pipeline is:

```
Page_Load_Start
  → HTML parsed, JS bundle evaluated (~50–100ms on fast connection)
  → React initializes, useState lazy initializer runs generate() (~5–15ms)
  → React renders full component tree with simulated data (~10–30ms)
  → First meaningful paint (all simulated sections visible)
  → useEffect hooks fire → API fetches initiated (non-blocking)
```

Total time from `Page_Load_Start` to first meaningful paint: approximately 65–145ms under normal conditions, well within the 500ms budget.

**API fetches never delay initial paint:** All three API clients (`FredIndicator`, `FxConversionInline`, `MarketauxNewsFeed`) initiate their fetches inside `useEffect`, which fires after the browser has painted. The Market Context Panel renders immediately with loading indicators for the API sections — the simulated data sections are never blocked.

---

### 7.3 50ms Main Thread Budget Per Tick (Req 14.5)

**Simulation generation is a one-time cost.** After initialization, the Simulator does not run again unless the user clicks "Refresh Data". There is no per-tick simulation work.

**Countdown timer tick cost:** On each 1-second tick, `CutOffTimeMonitor` calls `computeAllCutOffs(new Date())`, which performs:
- 6 calls to `secondsUntilCutOff()` (one per rail, plus 3 for ACH Same Day windows)
- Each call uses `Intl.DateTimeFormat.formatToParts()` — a fast native browser operation
- Total tick computation: < 1ms

The resulting state update (`setCutOffState`) triggers a re-render of `CutOffTimeMonitor` and `StatusBar` only (see Section 7.4). Both are lightweight components with no expensive child trees.

**Verifying the 50ms constraint during development:**

1. Open Chrome DevTools → Performance tab
2. Click Record, interact with the dashboard for 5–10 seconds, click Stop
3. In the flame chart, filter for "Timer Fired" events (these are the `setInterval` ticks)
4. Each tick should show a total scripting duration of < 1ms
5. For the initial load, find the "Evaluate Script" + "Render" block — total should be < 300ms
6. The "Long Tasks" overlay (red triangles) should show no tasks exceeding 50ms after initial load

---

### 7.4 Countdown Timer Without Full Re-Renders (Req 17.8)

The timer architecture ensures that only the components that actually need to update on each tick do so.

**Components that re-render on each tick:**
- `CutOffTimeMonitor` — owns the timer state
- `StatusBar` — reads `CutOffContext`, which is updated on each tick

**Components that do NOT re-render on each tick** (wrapped with `React.memo`):

```typescript
export const RailHealthCard        = React.memo(_RailHealthCard);
export const ExceptionGroupRow     = React.memo(_ExceptionGroupRow);
export const ExceptionDrillDown    = React.memo(_ExceptionDrillDown);
export const SettlementPositionTracker = React.memo(_SettlementPositionTracker);
export const SettlementTimeline    = React.memo(_SettlementTimeline);
export const FredIndicator         = React.memo(_FredIndicator);
export const FxConversionInline    = React.memo(_FxConversionInline);
export const MarketauxNewsFeed     = React.memo(_MarketauxNewsFeed);
```

These components receive props derived from `DataProvider` (which does not change on tick) or API state (which changes only on fetch completion). `React.memo` ensures they skip re-renders when their props are unchanged.

**Why `StatusBar` re-renders are acceptable:** `StatusBar` is a single horizontal bar with 4 text values. Its re-render cost is negligible — a few DOM text node updates. No child components of `StatusBar` are expensive.

**CutOffContext isolation:** `CutOffTimeMonitor` writes to `CutOffContext` on each tick. Only components that consume `CutOffContext` (currently only `StatusBar`) re-render as a result. Components that do not consume `CutOffContext` are unaffected regardless of `React.memo`.

### 7.5 Chart Performance (SettlementTimeline)

`SettlementTimeline` uses Recharts to render the intraday settlement timeline. Two optimizations prevent chart-related performance issues:

**Animation disabled:**

### 8.2 Error Message Matrix

```typescript
const MESSAGE_MAP: Record<ApiSource, Record<ApiErrorType, string>> = {
```

```typescript
function buildMessage(
  source: ApiSource,
  errorType: ApiErrorType,
  lastFetchedAt: Date | null
): string {
  const base = MESSAGE_MAP[source][errorType];
  if (!lastFetchedAt) return base;
  const formatted = lastFetchedAt.toLocaleTimeString('en-US', {
    hour: '2-digit', minute: '2-digit', timeZoneName: 'short',
  });
  return `${base} (Last updated: ${formatted})`;
}
```

### 8.3 ErrorState Component Behavior

```typescript
function ErrorState({
  source, errorType, lastFetchedAt, cachedDataAvailable,
  onRetry, isRetrying, children,
}: ErrorStateProps) {
  const message = buildMessage(source, errorType, lastFetchedAt);

  if (cachedDataAvailable) {
    // Show stale data with badge + retry button alongside
    return (
      <div className="error-state error-state--stale">
        <div className="stale-data-indicator">
          <span className="stale-badge">⚠ Stale data</span>
          <span className="stale-message">{message}</span>
          <RetryButton onRetry={onRetry} isRetrying={isRetrying} />
        </div>
        {children}  {/* stale cached data rendered normally */}
      </div>
    );
  }

  // No cache — show error message and retry only
  return (
    <div className="error-state error-state--empty">
      <span className="error-icon">⚠</span>
      <span className="error-message">{message}</span>
      <RetryButton onRetry={onRetry} isRetrying={isRetrying} />
    </div>
  );
}

function RetryButton({ onRetry, isRetrying }: { onRetry: () => void; isRetrying: boolean }) {
  return (
    <button
      className="retry-button"
      onClick={onRetry}
      disabled={isRetrying}
      aria-label="Retry"
    >
      {isRetrying ? <span className="spinner" aria-hidden="true" /> : 'Retry'}
    </button>
  );
}
```

**Retry isolation:** `onRetry` is a closure that calls only the fetch function for its own source. It does not trigger re-fetches of other API sections. Each API component manages its own `isRetrying` state independently.

**Retry flow:**
```typescript
const handleRetry = useCallback(async () => {

  try {
    await fetchData();   // source-specific fetch
  } finally {
    setIsRetrying(false);  // always reset, success or failure
  }
}, []);
```

### 8.5 Error Boundaries

Each of the three API section components is wrapped in a `ApiErrorBoundary`. If a component throws during render (e.g., due to unexpected data shape after a successful fetch), the boundary catches it and renders `ErrorState` with `errorType: 'malformed'`.

```typescript
interface ApiErrorBoundaryProps {
  source: ApiSource;
  lastFetchedAt: Date | null;
  onRetry: () => void;
  children: React.ReactNode;
}

interface ApiErrorBoundaryState {
  hasError: boolean;
}

class ApiErrorBoundary extends React.Component<
  ApiErrorBoundaryProps,
  ApiErrorBoundaryState
> {
  state: ApiErrorBoundaryState = { hasError: false };

  static getDerivedStateFromError(): ApiErrorBoundaryState {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: React.ErrorInfo) {
    console.error(`[RailWatch] ${this.props.source} render error:`, error, info);
  }

  render() {
    if (this.state.hasError) {
      return (
        <ErrorState
          source={this.props.source}
          errorType="malformed"
          lastFetchedAt={this.props.lastFetchedAt}
          cachedDataAvailable={false}
          onRetry={() => {
            this.setState({ hasError: false });
            this.props.onRetry();
          }}
          isRetrying={false}
        />
      );
    }
    return this.props.children;
  }
}
```
